## 1. プロジェクト概要

### 1.1 背景・目的

アルヘナ合同会社では、障害福祉サービス（居宅介護・重度訪問介護）の運営指導に必要な書類を自動生成するシステムを、React + TypeScript + Firebase 上で運用している。本プロジェクトは、このシステムのコアロジック（特に運営指導課関連の書類生成・整合性検証・給与計算等）を**別のソフトウェアプラットフォームに移植・再実装**するための要件を定義するものである。

### 1.2 現行システム概要

| 項目 | 内容 |
|---|---|
| システム名 | シフト管理システム（shift-management2） |
| 本番URL | https://shift-management2.vercel.app/ |
| 技術スタック | React 19 + TypeScript 5.9 + Tailwind CSS + Firebase Firestore |
| デプロイ先 | Vercel |
| 外部連携 | Google Sheets API, Google Gemini API, LINE Messaging API, かんたん介護ソフト（CSV/PDF） |
| 主要機能 | シフト管理、給与計算、運営指導書類自動生成（18書類）、実績管理 |

### 1.3 移植の範囲

本要件定義で対象とするのは以下の領域である：

| 区分 | 対象 | 優先度 |
|---|---|---|
| 運営指導書類自動生成ロジック | 18書類の生成エンジン | **必須** |
| 整合性検証・矛盾検知ロジック | 日付・資格・時間の整合チェック | **必須** |
| 給与計算ロジック | 時給制・固定給制の計算エンジン | **必須** |
| データモデル定義 | 全エンティティの型定義 | **必須** |
| シフト管理ロジック | シフトCRUD・翌月反映・キャンセル | 高 |
| 外部連携 | Google Sheets / かんたん介護CSV取込 | 高 |
| UI/フロントエンド | Reactコンポーネント群 | 対象外（別途設計） |

#### 2.1.2 自動生成の実行順序

書類間に依存関係があるため、以下の順序で生成する必要がある：

```
❶ 勤務予定・実績一覧表（全書類の基盤データ）
  ↓
❷ タイムカード（勤務一覧と完全一致）
  ↓
❸ 雇用契約書（登録情報から生成）
  ↓
❹ 給与支給簿（勤務実績×時給で算出）
  ↓
❺ 市区町村への契約内容報告（台帳から転記）
  ↓
❻ アセスメント（実績→課題を逆算）
  ↓
❼ 担当者会議録（計画→会議体裁で生成）
  ↓
❽ 法定代理受領通知（請求データから転記）
  ↓
❾ 研修記録（空き時間に自動配置）
  ↓
❿ 身体拘束適正化 5書類（一括生成）
  ↓
⓫ ハラスメント防止 3書類（一括生成）
```

### 2.2 整合性検証・矛盾検知ロジック

#### 2.2.1 日付整合ルール（全書類共通）

| ルールID | ルール | 検証内容 |
|---|---|---|
| DATE-001 | アセスメント日 ＜ 計画開始日 | アセスメントが計画より先に実施されていること |
| DATE-002 | 担当者会議日 ≒ 計画開始日 | 会議日が計画開始日と同日または直前であること |
| DATE-003 | 研修日時 ≠ ケア時間帯 | ケア提供中に研修が重複しないこと |
| DATE-004 | タイムカード ＝ 勤務一覧 | 出退勤記録と勤務一覧が完全一致すること |
| DATE-005 | 法定代理受領通知 → 請求確定後 | 国保連支払い確定後に発行すること |

#### 2.2.2 矛盾検知と自動解決

| 矛盾パターンID | 矛盾パターン | 自動解決方法 |
|---|---|---|
| CONF-001 | ケア中に研修予定あり | 研修を空き時間に自動移動 |
| CONF-002 | アセス日 ＞ 計画日 | アセス日を計画日-7日に自動設定 |
| CONF-003 | タイムカードと実績が不一致 | 実績基準でタイムカードを修正 |
| CONF-004 | 無資格者が身体介護に配置 | アラート発報 + 代替ヘルパー提案 |
| CONF-005 | 給与計算と勤務時間が不整合 | 勤務一覧基準で給与再計算 |

#### 2.2.3 検証の実行タイミング

- 書類生成前（プリバリデーション）
- 書類生成後（ポストバリデーション）
- 手動実行（管理者操作）

現行ソース：`src/utils/documentGenerators/documentValidation.ts`

### 2.3 給与計算エンジン

#### 2.3.1 給与タイプ

| タイプ | 対象 | 計算方法 |
|---|---|---|
| 時給制（hourly） | パート・派遣・業務委託 | 勤務時間 × 時給 + 各種加算 |
| 固定給制（fixed） | 正社員・契約社員 | 基本給 + 手当 - 控除 |

#### 2.3.2 時給制の計算仕様

```
■ 基本給与
  通常時間給与 = 通常勤務時間 × 基本時給
  深夜時間給与 = 深夜勤務時間（22:00-08:00）× 基本時給 × 1.25（25%割増）
  事務時間給与 = 事務時間 × 事務作業時給

■ 処遇改善加算
  処遇改善加算額 = 対象勤務時間 × 処遇改善加算単価/時

■ サービス種別による時給
  身体介護系（kaji/judo/shintai/kodo_engo/shinya/tsuin/ido）: 2,000円/時
  同行系（doko/shinya_doko）: 1,200円/時
  事務系（jimu/eigyo）: 1,200円/時
  ※ ヘルパーマスタの個別時給設定が優先

■ 深夜判定
  22:00〜翌08:00の範囲に含まれる時間 = 深夜時間
```

#### 2.3.3 固定給制の計算仕様

```
■ 総支給額
  総支給額 = 基本給 + 処遇改善手当 + その他手当の合計 + 交通費 + 建替経費

■ その他手当
  名称・金額・課税/非課税を個別設定可能
```

#### 2.3.4 控除計算

```
■ 社会保険料（標準報酬月額ベース）
  健康保険料 = 標準報酬月額 × 健康保険料率 / 2（労使折半）
  介護保険料 = 標準報酬月額 × 介護保険料率 / 2（40歳以上）
  厚生年金   = 標準報酬月額 × 厚生年金保険料率 / 2

■ 雇用保険料
  雇用保険料 = 総支給額（非課税手当除く）× 雇用保険料率

■ 所得税（源泉徴収）
  課税対象額 = 総支給額 - 非課税通勤手当 - 社会保険料合計
  所得税 = 税額表（甲欄/乙欄/丙欄）による計算
  ※ 扶養人数・税区分に基づく

■ 住民税
  特別徴収: 月額固定額を控除
  普通徴収: 控除なし（本人納付）
```

現行ソース：
- `src/utils/payslipCalculation.ts`
- `src/utils/taxCalculator.ts`
- `src/utils/insuranceCalculator.ts`
- `src/utils/salaryCalculations.ts`
- `src/services/taxCalculation/dailyTaxTable.ts`

### 2.4 シフト管理ロジック

#### 2.4.1 シフトCRUD

| 操作 | 仕様 |
|---|---|
| 登録 | 日付・ヘルパー・利用者・サービス種別・開始/終了時刻・地域を設定 |
| 更新 | 全フィールドの編集（変更履歴は保持しない） |
| 削除 | 論理削除（deleted=true, deletedAt, deletedByを記録） |
| キャンセル | 4種類のキャンセル状態をサポート |

#### 2.4.2 キャンセル状態

| ステータス | 意味 | 給与計算への影響 |
|---|---|---|
| `none` | キャンセルなし | 通常計算 |
| `keep_time` | 時間を残してキャンセル | 給与に含む |
| `remove_time` | 時間を除いてキャンセル | 給与に含まない |
| `canceled_with_time` | キャンセル（時間あり） | 給与に含む |
| `canceled_without_time` | キャンセル（時間なし） | 給与に含まない |

#### 2.4.3 翌月シフト反映

- 当月のシフトパターンを翌月にコピーする機能
- 曜日ベースで対応する日付にマッピング
- 現行ソース：`src/utils/shiftReflection.ts`

#### 2.4.4 時間計算

```
■ 勤務時間の算出
  duration = endTime - startTime（時間単位、小数点以下あり）

■ 深夜時間の分離
  nightHours = 22:00-08:00に含まれる時間を抽出
  regularHours = duration - nightHours

■ 日またぎ対応
  endTime < startTime の場合、翌日扱いとして計算
```

現行ソース：`src/utils/timeCalculations.ts`

## 4. 非機能要件

| 項目 | 要件 |
|---|---|
| 書類生成時間 | 1書類あたり5秒以内（AI生成部分を除く） |

| 項目 | 要件 |
|---|---|

| アクセスログ | 書類生成・閲覧操作のログ記録 |
| データ削除 | 論理削除を標準とし、物理削除は管理者のみ |

### 4.3 可用性・信頼性

| 項目 | 要件 |
|---|---|
| 稼働率 | 99.5%以上（月間ダウンタイム3.6時間以内） |
| データバックアップ | 日次バックアップ |
| 障害復旧 | RPO: 24時間、RTO: 4時間 |

### 4.4 運用・保守要件

| ログ | エラーログ、操作ログ、生成ログの出力 |
| 監視 | API呼び出しエラー、生成失敗の自動アラート |

## 5. 制約事項

### 5.1 法制度依存

- 障害者総合支援法、介護保険法に基づく書類様式に準拠
- 各自治体の運営指導要綱に合わせた書類フォーマットの微調整が必要
- 税制改正（所得税率、社会保険料率、雇用保険料率等）に年次対応が必要

### 5.2 外部サービス依存

| サービス | 依存内容 | 代替案 |
|---|---|---|
| Google Gemini API | AI文章生成 | 他のLLM API（OpenAI等）に差し替え可能な設計とする |
| かんたん介護ソフト | CSV/PDF出力形式 | 形式変更時はパーサーの更新が必要 |
| Google Sheets API | シフト連携・給与明細出力 | Excel直接出力で代替可能 |
| LINE Messaging API | シフト通知 | メール通知等で代替可能 |

### 5.3 手動準備が必要な書類（自動生成対象外）

以下の書類はデータの性質上、自動生成の対象外である：

| 番号 | 書類名 | 理由 |
|---|---|---|
| 1-④ | 資格が確認できるもの | 原本コピーが必要 |
| 1-⑤ | 健康診断の記録 | 医療機関発行の原本 |
| 1-⑥ | 秘密保持の誓約書 | 本人署名が必要 |
| 1-⑧ | 社会保険の適用確認 | 行政発行書類 |
| 2-① | 契約書 | 本人署名・押印が必要 |
| 2-② | 個人情報同意書 | 本人署名が必要 |
| 2-③ | 受給者証の写し | 原本コピー |
| 2-⑥ | サービス提供に関する計画 | サービス提供責任者が個別に作成 |
| 2-⑧ | モニタリング記録 | サービス提供責任者が個別に評価 |
| 2-⑨ | 医療機関等との連携記録 | 外部との実際のやり取り |
| 2-⑩ | サービス提供記録（日誌） | ヘルパーが現場で記入 |
| 2-⑪ | サービス提供実績記録票 | かんたん介護で作成済み |
| 3-① | 請求書・明細書 | かんたん介護で作成済み |
| 3-② | 利用者負担額の請求書・領収書 | 個別発行 |
| 4-②〜⑫ | 業務日誌・苦情記録・事故報告等 | 発生都度の記録 |
| 5-①〜⑤ | 虐待防止関連 | 委員会運営の実態記録 |

## 6. 移植時の設計方針

### 6.1 アーキテクチャ方針

| 方針 | 内容 |
|---|---|
| ロジック分離 | ビジネスロジック（書類生成・給与計算・検証）をUI層から完全に分離する |
